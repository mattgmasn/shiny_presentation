What do we mean by interactivity?

  • This tutorial takes a fairly broad view - we include any feature that enables data or visualisations to be changed based on user input

  • This could be as simple as zooming in on a chart, or as large as a fully interactive dashboard with multiple datasets

  • This tutorial only touches on the basics, but these can be developed into building complex and interesting tools to really showcase your data and findings

Why would I want to make my data or findings interactive?

  • Engagement Interactivity can help data and analysis become more accessible and understandable, especially for non-academic audiences
  • Efficiency Interactive visualisations can save on space when you want to display a lot of information
  • Usability Interactive tools can help automate tasks and processes
  • Exploration Interactive tools can make data exploration easier and faster
  • For fun!

Options for interactivity in R

  • Table/chart-level interactives e.g., R packages such as plotly, DT and flowmapper

  • Reports or webpages e.g., by using Quarto to publish R code and analysis

  • Dashboards/interactive tools e.g., using R Shiny to allow for changes based on user inputs

Chart level interactives - DT

  • DT is an R package that lets you create interactive data tables that can be exported as html objects using very simple R code - you can embed these in quarto documents or presentations

  • This is pretty limited as fair as interactivity goes, but it does allow for quicker searching through data and for your audience to be able to look at an entire dataset easily

Chart level interactives - plotly

  • Plotly is an R package that lets you create interactive charts that can be exported as html objects using very simple R code - you can embed these in quarto documents or presentations

  • It also can be used to convert existing charts made with ggplot interactive

  • Interactivity with plotly is fairly limited, but can be used to allow users to zoom in on charts, highlight sections and display more information in tooltips as they hover over parts of the chart

Plotly example - code

# First load your packages
library(ggplot2)
library(plotly)

# Create an interactive chart using plotly directly
plot_plotly <- plot_ly(data = iris, 
        x = ~Sepal.Length, 
        y = ~Petal.Length, 
        color = ~Species)

# Or, apply ggplotly to an existing ggplot chart
plot_ggplot <- ggplot() +
  geom_point(data = iris, 
             aes(x = Sepal.Length, 
                 y = Petal.Length, 
                 color = Species))

ggplotly(plot_ggplot)

Plotly example - customising tooltips

  • How can we make the interactive features more useful?

  • At the minute the tooltips aren’t very helpful but we could change that using some simple R code

# Create custom tooltip text
iris$tooltip_text <- paste(
  "Species:", iris$Species, "<br>",
  "Sepal Length:", iris$Sepal.Length, "cm", "<br>",
  "Petal Length:", iris$Petal.Length, "cm")

# Add tooltip to the plotly plot
plot_plotly_tooltips <- plot_ly(
  data = iris,
  x = ~Sepal.Length,
  y = ~Petal.Length,
  color = ~Species,
  text = ~tooltip_text,         # custom hover text
  hoverinfo = "text")          # only show text (not x/y by default)

R Shiny

  • R Shiny is one of the most used tools in R for creating dashboards to showcase your data and analysis

  • Shiny can be used in both R and Python

  • Shiny apps that are developed can be deployed onto webpages - there is even a dedicated server where Shiny apps can be hosted for free (with limits!)

R Shiny - pros

  • Flexible and can utilise R functions when working with data and creating visualisations

  • Supports complex interactive elements

  • Allows for user inputs to change outputs

  • Lots of documentation available

  • Can be hosted as a webpage on shinyapps.io for free

R Shiny - cons

  • Can be a steep learning curve

  • Not suitable for very large datasets without performance issues

  • Hosting apps for a more general audience on shinyapps.io will cost money

R Shiny - components

Shiny apps have two components:

  • The user interface (ui):

    • This controls the appearance and layout of your applications.

  • The server:

    • This is where you will manipulate the data, create and define your visualisations to be called by the UI

Both need to be present for your dashboard to work!

R Shiny - setting up

  • The ui and server components can either be in the same R script or in separate files in the same folder

  • Alternatively, in RStudio you can specifically create a R Shiny web app file when you create a new file

Simple example - ui

  • Let’s make a very simple Shiny app using the gapminder dataset to look at life expectancy over time by country
# Load packages
library(tidyverse)
library(shiny)
library(gapminder)

ui <- fluidPage(
  titlePanel("📈 Life Expectancy Over Time"),
  selectInput("country", "Select a country:", choices = unique(gapminder$country)),
  plotOutput("lifePlot")
)
  • Everything is in a fluidPage to help format the app’s layout
  • We have an input (a dropdown - “country”) and an output (a plot - “lifePlot”)

Simple example - server

server <- function(input, output, session) {
  output$lifePlot <- renderPlot({
    gapminder %>%
      filter(country == input$country) %>%
      ggplot(aes(x = year, y = lifeExp)) +
      geom_line(color = "steelblue", linewidth = 1.5) +
      geom_point(color = "darkred", size = 2) +
      labs(
        title = paste("Life Expectancy in", input$country),
        x = "Year",
        y = "Life Expectancy (years)"
      ) +
      theme_minimal()
  })
}

  • The server component sits inside a function

  • Our output here is a plot the data for which is filtered based on the input in the ui (“country”)

Simple example - output

  • Once you have the ui and server components (either in one R script or in seperate ones in the same folder) you simply run the below code to see your app

    # Run the app
shinyApp(ui, server)

Other options for Shiny apps

  • Different inputs - dropdown selectors, but also sliders, buttons or check boxes to select inputs

  • Different outputs - charts and plots, but also tables and download buttons for users to output the data they have selected

Adding complexity - server

server <- function(input, output, session) {
  # Filtered data for plots that is reactive to inputs
  filtered_data <- reactive({
    gapminder %>%
      filter(year == input$year, continent %in% input$continent)
  })
  
  # Interactive bubble plot
  output$bubblePlot <- renderPlotly({
    p <- ggplot(filtered_data(), aes(
      x = gdpPercap,
      y = lifeExp,
      size = pop,
      color = continent,
      text = paste0("Country: ", country,
                    "<br>GDP per Capita: $", round(gdpPercap, 0),
                    "<br>Life Expectancy: ", round(lifeExp, 1), " years",
                    "<br>Population: ", format(pop, big.mark = ","))
    )) +
      geom_point(alpha = 0.7) +
      scale_x_log10() +
      theme_minimal() +
      labs(title = paste("Year:", input$year),
           x = "GDP per Capita (log scale)",
           y = "Life Expectancy (years)")
    
    ggplotly(p, tooltip = "text")
  })
  
  # Summary table
  output$summaryTable <- renderTable({
    filtered_data() %>%
      group_by(continent) %>%
      summarise(
        Countries = n(),
        Avg_LifeExp = round(mean(lifeExp), 1),
        Avg_GDP = round(mean(gdpPercap), 0),
        Total_Pop = sum(pop)
      )
  })
  
  # Download handler
  output$download <- downloadHandler(
    filename = function() {
      paste0("gapminder_", input$year, ".csv")
    },
    content = function(file) {
      write.csv(filtered_data(), file, row.names = FALSE)
    }
  )
}

Hosting your Shiny apps

  • Now you have an app it’s time to put it somewhere!

  • You don’t have to, but having an easy url to your app can make it easier to use rather than having to run it via R each time

  • Most of the time, for any apps you create for yourself or for a small team, deploying your apps on shinyapps.io is the best option

  • A guide for this can be found here

  • There are limits on the number of apps you can deploy and the amount of time they can be used per month, but in most cases this is enough
